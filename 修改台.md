## 2026-04-02 后端掉线修复记录

### 现象

- 页面右下角反复出现“无法连接到标书后端服务”提示。
- 实际检查结果为 `127.0.0.1:3001` 未监听，不是前端误报。

### 直接原因

- 后端之前是通过 `start-stack.ps1` 直接拉起单个 Node 进程。
- 该进程一旦异常退出，`3001` 会直接失效，前端所有保存、生成、解析相关请求都会报后端不可用。

### 为什么之前看不到原因

- 旧链路只有启动日志，没有进程级退出日志。
- 所以之前最多只能看到“Nest application successfully started”，看不到它后面是因异常退出、被终止，还是其他原因结束。
- 同时旧的 `backend.pid` 会残留，容易让人误判成“后端应该还活着”。

### 本次修复

- 新增 `project/backend/scripts/backend-supervisor.ps1`
  - 负责守护后端主进程
  - 发现异常退出后自动重启
  - 记录退出时间、退出码
  - 归档最近一轮 `backend.out.log` / `backend.err.log`
- 修改 `project/backend/scripts/start-stack.ps1`
  - 不再直接裸跑 Node
  - 改为启动 supervisor
  - 启动前清理失效 supervisor / stale pid
- 修改 `project/backend/scripts/stop-stack.ps1`
  - 同时停止 backend 子进程与 supervisor
  - 增加 stop flag，避免停服务时被 supervisor 立即拉起
- 修改 `project/backend/src/main.ts`
  - 增加 `uncaughtException`
  - 增加 `unhandledRejection`
  - 增加 `SIGINT` / `SIGTERM`
  - 增加 `beforeExit` / `exit`
  - 以后如果再次退出，日志里会明确留下退出线索

### 当前判断

- 这次真实根因是“后端进程已经退出，但原启动方式没有守护和退出记录”，不是前端页面本身坏掉。
- 之前无法精确还原第一次退出的业务触发点，这是因为旧版本在退出前没有把原因打到日志里。

### 验证结果

- 已重新启动后端，`127.0.0.1:3001` 恢复监听。
- 已验证 `POST /api/auth/login` 返回 `201`。
- 已人工终止一次后端子进程，supervisor 能在数秒内自动拉起新进程并恢复服务。
