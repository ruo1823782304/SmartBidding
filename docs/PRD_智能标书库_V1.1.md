### 1.3 名词定义

| 名词 | 定义 |
|------|------|
| 招标文件 | 用户上传的原始招标要求文件，支持 PDF、Word、TXT 等 |
| 标书解析 | 对招标文件进行 AI 结构化分析，提取资格要求、评分点、废标条款等 |
| 应标大纲 | 根据招标要求与公司习惯结构生成的标书章节树，作为后续编制导航 |
| 标书编制 | 按项目应标大纲撰写、复用、生成、审改标书内容的过程 |
| 企业库 | 存储并管理资质、业绩、技术方案、人员简历、历史标书片段等资料库 |
| 历史标书入库 | 批量上传已完成标书，由 AI 拆解、归类、打标签并形成可复用素材 |
| 片段库 | 从历史标书或资料中抽取出的可复用内容单元，可用于推荐或插入 |
| 项目 | 系统内的一级业务实体，对应一次具体投标任务 |
| 章节任务 | 针对项目中某个或某组章节分配给具体负责人的任务单元 |
| 归档 | 项目结束后将标书、招标文件、复盘、对手信息统一沉淀 |

## 二、产品概述

### 2.1 产品定位

智能标书库是一套面向中小型投标团队的标书全生命周期管理系统，覆盖：

**资料沉淀 → 招标解析 → 应标大纲生成 → 标书编制 → 协作审批 → 导出交付 → 归档复盘**

系统通过企业库与历史标书入库能力，持续沉淀资质、业绩、技术方案、人员简历和历史表达片段；在收到新招标要求时，通过 AI 分析快速产出可编辑应标大纲，并在编制阶段按章节推荐可复用资料或生成草稿，最终按公司模板输出 Word / PDF 标书成品，并在项目结束后完成归档与复盘，形成可持续复用的投标知识资产。

### 2.2 核心价值

| 价值点 | 说明 |
|--------|------|
| 招标即大纲 | 招标文件解析后直接生成应标大纲，减少人工拆目录与对齐成本 |
| 资料持续沉淀 | 历史标书、资质、简历、业绩、方案统一入库，越用越完整 |
| AI 辅助而非黑盒替代 | AI 负责提取、归类、推荐、生成草稿，关键节点由人工确认 |
| 多人协作可控 | 章节级任务分配、只读/可编辑权限、初审复审与驳回闭环 |
| 输出可交付 | 依据公司模板和标题规范生成正式标书成品 |
| 结果可复盘 | 形成项目归档、复盘记录、竞争对手档案和可统计数据 |

### 2.3 目标用户

- 投标负责人 / 项目经理：创建项目、统筹进度、分配章节、协调审查
- 技术标制作人：负责技术标相关章节编写与提交
- 商务标制作人：负责商务标、资信标相关章节编写与提交
- 机动人员：协助资料上传、部分章节补充、资料整理
- 初审人：执行初审，关注完整性、合规性、格式性问题
- 复审人：执行终审，确认整体质量与提交版本
- 管理员：管理人员、角色、系统参数、模型配置、审计能力

## 三、业务目标与成功指标

### 3.1 业务目标

1. 降低新标书从读标到成稿的时间成本  
2. 提升历史资料复用率，减少重复撰写  
3. 提高多人协作下的进度透明度和责任清晰度  
4. 规范导出成品质量，降低人工排版与返工  
5. 沉淀投标过程数据，为后续决策和复盘提供依据  

### 3.2 成功指标（建议上线后跟踪）

| 指标 | 目标建议 |
|------|----------|
| 招标解析后生成首版应标大纲时长 | 5 分钟内 |
| 历史资料 / 片段复用率 | ≥ 40% |
| AI 生成大纲一次采纳率 | ≥ 70% |
| 标书导出后人工排版调整时长 | 降低 50% |
| 章节任务按时完成率 | ≥ 85% |
| 审批平均流转时长 | 较线下流程降低 30% |
| 项目归档完成率 | ≥ 95% |
| 复盘填写完成率 | ≥ 80% |

### 3.3 项目边界

本期纳入范围：

- 招标文件上传与结构化解析
- 应标大纲生成、编辑与同步
- 标书按章节编制、复用、AI 生成、保存
- 企业库资料管理与历史标书智能归类
- 项目协作、章节分配、审批流与导出
- 项目归档、复盘、竞争对手档案、数据看板

本期不纳入范围：

- 对外投标平台自动提交
- 电子签章 / CA 证书集成
- 财务预算、合同执行、售后履约等后链路业务
- 复杂多组织多租户隔离能力

## 四、系统结构与导航

### 4.1 一级导航

| 区域 | 菜单项 | 说明 |
|------|--------|------|
| MENU | 标书解析 | 上传招标文件、AI 解析、生成应标大纲 |
| MENU | 标书编制 | 基于项目应标大纲编写标书，支持复用与 AI 生成 |
| MENU | 企业库 | 管理资质、业绩、技术方案、简历、历史标书、片段库 |
| MENU | 项目协作 | 项目看板、任务分配、审批流、进度追踪 |
| MENU | 数据中心 | 经营看板、知识库、复盘、竞争对手、项目归档 |
| ACCOUNT | 个人中心 | 我的任务、我负责的章节、待办审批、我的项目 |
| ACCOUNT | 编辑资料 | 修改个人信息、密码、联系方式 |
| OTHER | 设置（概念层，当前实现已拆分） | 人员配置、角色、上下级、模型配置、导出模板、系统参数 |
| OTHER | 人员配置 | 管理员维护用户、角色、上下级、账号状态 |
| OTHER | 模型配置 | 管理员维护供应商 API-Key、Coding Plan 解析与模型生效 |
| OTHER | 帮助 | 使用说明、FAQ、支持方式 |

### 4.2 一级业务实体

系统围绕“项目”组织全部业务数据：

- 一个项目对应一次招标任务
- 一个项目绑定一份或多份招标文件
- 一个项目拥有一套应标大纲
- 一个项目下存在多个章节任务与审批记录
- 一个项目最终形成一份或多份导出成品、复盘记录和归档记录

### 4.3 当前项目上下文

所有以下操作均需绑定当前项目：

- 招标解析结果查看
- 应标大纲生成与编辑
- 标书编制内容写入
- 章节任务分配
- 审批流提交与流转
- 导出与归档

若用户未选择项目，系统应提示先“创建项目”或“选择项目”，不得将解析结果写入默认上下文。

## 五、角色与权限

### 5.1 角色定义

| 角色 | 职责说明 |
|------|----------|
| 管理员 | 维护用户、角色、权限、上下级关系、系统参数与模板配置；全局可见 |
| 投标负责人 / 项目经理 | 创建项目、分配章节任务、查看全项目进度、推进审批 |
| 技术标制作人 | 编写技术标相关章节，处理被分配的技术章节任务 |
| 商务标制作人 | 编写商务标、资信标相关章节，处理被分配任务 |
| 机动人员 | 协助资料上传、编辑被分配的补充章节、处理辅助任务 |
| 初审人 | 对已提交版本执行初审，可通过或驳回并给出意见 |
| 复审人 | 对初审通过版本执行终审，可通过或驳回并给出意见 |

说明：

- 同一用户默认绑定一个角色
- 如需兼岗，可通过项目任务和审批分配实现，不强制新增角色类型

### 5.2 页面权限

| 菜单 / 能力 | 管理员 | 项目经理 | 技术标制作人 | 商务标制作人 | 机动人员 | 初审人 | 复审人 |
|-------------|--------|----------|--------------|--------------|----------|--------|--------|
| 标书解析 | ✓ | ✓ | ✓ | ✓ | ✓ | — | — |
| 标书编制 | ✓ | ✓ | ✓ | ✓ | ✓ | 只读可选 | 只读可选 |
| 企业库 | ✓ | ✓ | ✓ | ✓ | ✓ | — | — |
| 项目协作 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| 数据中心 | ✓ | ✓ | ✓ | ✓ | ✓ | 只读可选 | 只读可选 |
| 设置 | ✓ | — | — | — | — | — | — |
| 导出正式成品 | ✓ | ✓ | 按权限 | 按权限 | 按权限 | — | ✓ |

### 5.3 章节级权限规则

章节是否可编辑，由以下条件共同决定：

- 用户角色具备编制权限
- 当前用户被分配该章节任务
- 当前章节未处于锁定状态
- 当前项目处于可编辑阶段

补充规则：

- 管理员与项目经理可在授权范围内查看并干预全部章节
- 初审人、复审人默认只读查看章节内容，不直接改正文
- 若章节已提交审批，章节默认锁定；驳回后重新打开编辑
- 系统应记录章节最后编辑人、最后编辑时间、当前版本号

### 5.4 上下级关系

- 用于组织关系展示、统计汇总和后续审批规则扩展
- 不直接改变角色权限
- 审批人可按项目单独指定，不强依赖上下级链路

## 六、端到端业务流程

### 6.1 总流程

```text
资料沉淀
  ├─ 上传资质 / 业绩 / 技术方案 / 简历
  └─ 历史标书批量入库 → AI 拆解归类 → 写入企业库 / 片段库

收到新招标
  ├─ 创建项目
  ├─ 上传招标文件
  ├─ AI 解析招标要求
  └─ 生成应标大纲 → 人工微调确认

标书编制
  ├─ 章节导航 = 项目应标大纲
  ├─ 分配章节负责人
  ├─ 按章节复用历史资料 / AI 生成草稿
  ├─ 编辑、保存、版本记录
  └─ 全部完成后提交审批

协作审批
  ├─ 初审
  ├─ 驳回后回到具体章节负责人
  ├─ 复审
  └─ 审批通过后进入导出

导出与归档
  ├─ 选择模板、标题规则、导出版本
  ├─ 输出 Word / PDF
  ├─ 填写复盘
  └─ 归档标书、招标文件、复盘、对手信息
```

### 6.2 关键原则

- 以“项目”为主线组织全链路
- 以“应标大纲”为编制导航和任务分配依据
- AI 输出需可追溯、可确认、可修改、可回写
- 历史沉淀必须可被后续项目检索、推荐、复用

## 七、AI 能力定义

### 7.1 AI 能力总原则

- AI 只做辅助，不直接替代人工决策
- 所有关键 AI 结果需支持人工确认或修订
- 结构化结果优先于长文本结果
- 重要结论尽可能保留来源定位信息
- 修订后的人工结果应可回写，作为后续推荐依据

### 7.2 AI 场景定义

| 场景 | 输入 | 输出 | 人工确认点 |
|------|------|------|------------|
| 招标文件解析 | 招标文件全文 | 资格要求、评分点、废标条款、提交材料、关键节点、风险提示、来源定位 | 解析结果确认 |
| 应标大纲生成 | 招标解析结果 + 公司模板偏好 | 一/二/三级章节树、章节类型、建议负责人类型 | 大纲确认与编辑 |
| 历史标书智能归类 | 多份历史标书 | 相似维度标签、可复用章节、可抽取片段、归类结果 | 归类结果确认 |
| 资质 / 简历抽取 | 资质文件、简历文件 | 关键信息字段、标签、建议适用章节 | 入库前确认 |
| 章节素材推荐 | 当前章节标题 + 章节类型 + 项目要求 | 推荐资质、业绩、简历、历史片段、技术方案 | 插入前确认 |
| 章节草稿生成 | 当前章节要求 + 可用素材 + 公司表达偏好 | 章节草稿正文 | 采纳、编辑、弃用 |
| 复盘摘要 | 项目结果 + 审批记录 + 人工输入 | 复盘摘要、改进建议、风险点 | 保存前确认 |

### 7.3 AI 输出要求

1. 招标解析结果需支持结构化字段展示  
2. 涉及结论性信息时应附来源定位，至少支持页码、段落或原文引用片段  
3. 应标大纲必须为结构化树，不允许仅输出一段普通文本  
4. 历史标书归类结果需可落到“分类资料”或“片段库”  
5. 章节草稿生成需支持选择语气、长度、引用素材范围  
6. 招标解析结果应支持分段落库与分段输出，不要求整份文件全部完成后才可查看首个主类结果  
7. 招标解析应采用“规则匹配 + 模型补全 + 模型校验”的混合方式，避免完全依赖单次模型输出

## 八、功能需求

### 8.1 项目管理

#### 8.1.1 功能目标

为每次投标建立独立项目，作为招标文件、应标大纲、标书内容、任务、审批和归档的统一容器。

#### 8.1.2 功能要求

- 支持创建、编辑、归档项目
- 项目字段至少包括：项目名称、项目编号、招标单位、截止时间、项目类型、负责人、状态
- 支持从首页、项目协作、标书解析、标书编制等页面切换当前项目
- 从首页项目卡片进入其他页面时，需自动写入当前项目上下文

#### 8.1.3 状态

项目状态建议包括：

- 待开始
- 解析中
- 编制中
- 待初审
- 待复审
- 待导出
- 已完成
- 已归档

---

### 8.2 标书解析

#### 8.2.1 功能目标

将原始招标文件转换为结构化、可追溯、可复用的读标结果，为应标大纲生成和标书编制提供依据。

#### 8.2.2 功能要求

- 支持上传 PDF、Word、TXT
- 支持单文件或多文件上传
- 当前实现校准：P0 以单文件上传并发起单次解析为准，多文件合并解析暂不作为当前验收范围
- AI 自动提取以下内容：
  - 资格要求
  - 评审标准与评分项
  - 废标 / 无效条款
  - 投标文件组成要求
  - 关键时间节点
  - 特殊格式要求
  - 风险提示
- 解析结果顶层按以下 7 个主类组织：
  - 基础信息
  - 资格要求
  - 评审要求
  - 投标文件要求
  - 无效标与废标项
  - 应标需提交文件
  - 招标文件审查
- 解析结果应结构化展示，可分页签、分组或标签化查看
- 每一条解析结论应支持查看来源信息
- 解析任务应支持“整体进度 + 主类进度”双层状态展示
- 每个主类状态至少包括：pending / running / completed / failed
- 已完成的主类应立即可查看，不必等待整份文件全部解析完成
- 当前解析实现应支持将已完成主类的内容先行返回前端，供用户逐类查看
- 支持人工修正解析结果并保存
- 解析完成后可触发“生成应标大纲”

#### 8.2.3 验收标准

- 上传招标文件后可完成一次解析任务并展示结构化结果
- 解析结果可绑定到当前项目，不丢失
- 至少支持对资格要求、评分项、废标条款进行结构化展示
- 解析过程中，七大主类按钮可展示待处理、进行中、已完成、失败状态
- 任一主类完成后，用户可在该主类按钮变为完成态后立即进入查看
- 用户可对解析结果手动编辑并保存

---

### 8.3 应标大纲管理

#### 8.3.1 功能目标

根据招标要求和公司模板规则，生成用于写标书的章节大纲，并作为编制导航、任务分配和导出目录的统一来源。

#### 8.3.2 功能要求

- 基于解析结果生成应标大纲
- 大纲结构至少支持三级标题
- 每个章节节点包含：
  - 章节 ID
  - 标题
  - 层级
  - 排序
  - 所属类别
  - 建议负责人类型
  - 是否必填
- 支持增删改章节、拖拽排序、移动层级
- 大纲确认后同步到标书编制页左侧导航
- 大纲修改后应同步影响：
  - 章节导航
  - 可分配章节列表
  - 导出目录结构
- 已分配章节被删除或重命名时，应给出提示并要求管理员确认影响

#### 8.3.3 验收标准

- 招标解析完成后可一键生成结构化应标大纲
- 大纲编辑后在标书编制页实时生效
- 默认模板与 AI 大纲均可用，且优先级可控

---

### 8.4 标书编制

#### 8.4.1 功能目标

让用户基于项目应标大纲逐章编写标书，并通过历史资料复用和 AI 生成提升编制效率。

#### 8.4.2 功能要求

- 左侧导航必须来源于当前项目应标大纲
- 章节编辑区支持：
  - 富文本
  - 表格
  - 图片
  - 附件引用
- 每个章节提供两个主要入口：
  - 从历史 / 企业库复用
  - AI 生成草稿
- 支持自动推荐当前章节可用素材：
  - 资质
  - 业绩
  - 技术方案
  - 人员简历
  - 历史片段
- 用户可将推荐素材一键插入或仅作参考
- 支持自动保存或定时保存
- 支持章节版本记录和版本恢复
- 支持章节“标记完成”
- 支持全书预览与目录预览

#### 8.4.3 协作规则

- 用户仅可编辑被分配章节
- 同一章节默认单人编辑，避免冲突
- 若需多人协作同一章节，应由管理员开启协同模式并保留版本记录
- 审批中章节默认锁定
- 驳回后仅打开被驳回章节或整书重新编辑，规则由审批人选择

#### 8.4.4 验收标准

- 章节导航与项目大纲一致
- 用户仅可编辑有权限的章节
- 每章节均可执行“复用历史”和“AI 生成”
- 保存后刷新不丢内容
- 支持查看章节历史版本

---

### 8.5 企业库

#### 8.5.1 功能目标

统一管理投标所需资料，作为 AI 推荐和人工复用的基础数据源。

#### 8.5.2 分类

- 企业资质
- 项目业绩
- 技术方案
- 中标案例
- 人员简历
- 标书归档
- 历史片段库

#### 8.5.3 功能要求

- 支持上传文件、录入文本、编辑元数据、打标签
- 支持按分类、标签、时间、适用行业、适用项目类型筛选
- 支持资料与章节类型建立关联
- 支持简历和资质的字段化提取
- 支持技术方案和案例建立摘要与关键词
- 支持将资料标记为“推荐优先”“禁用”“已过期”

#### 8.5.4 验收标准

- 企业库各分类可单独管理
- 资料可被章节推荐正确检索到
- 资质和简历至少支持一类结构化抽取

---

### 8.6 历史标书入库与智能归类

#### 8.6.1 功能目标

支持批量上传历史标书，通过 AI 识别相似结构和可复用内容，将其沉淀为企业库资料与可复用片段库。

#### 8.6.2 功能要求

- 支持批量上传多份历史标书
- AI 对每份标书执行以下处理：
  - 识别目录结构
  - 拆分章节与片段
  - 提取共性维度
  - 打标签
  - 识别适用场景
- 共性维度至少包括：
  - 资质类
  - 业绩类
  - 人员类
  - 技术方案类
  - 商务承诺类
  - 格式模板类
- 支持将归类结果写入：
  - 企业库对应分类
  - 片段库
  - 历史标书索引库
- 支持人工确认、修正、合并、删除归类结果
- 支持查看某段内容来源于哪份历史标书、哪一章节
- 当前实现校准：历史标书批量归类接口当前仅提供任务入口占位，不应作为 2026-03-31 已打通能力对外承诺

#### 8.6.3 验收标准

- 支持上传多份历史标书并输出归类结果
- 归类结果不是纯文本摘要，而是可检索、可确认的数据项
- 归类结果可用于后续章节推荐
- 当前阶段不将“批量历史标书真实归类完成”纳入 P0 已完成验收项

---

### 8.7 项目协作与审批

#### 8.7.1 功能目标

让多人围绕同一项目协作完成标书编制，并通过明确任务和审批流程保证进度与质量。

#### 8.7.2 功能要求

- 支持项目看板，查看全部项目状态
- 支持按章节分配负责人
- 任务分配时应写入持久化章节权限，不得仅保存在前端状态
- 个人中心展示：
  - 我的任务
  - 我负责的章节
  - 待办审批
  - 即将到期任务
- 支持章节或整书提交审批
- 初审支持通过 / 驳回 / 指定修改章节
- 复审支持通过 / 驳回 / 退回初审或编制阶段
- 审批意见必须留痕

#### 8.7.3 状态流转

章节状态建议包括：

- 待开始
- 编写中
- 已完成待提交
- 待初审
- 初审驳回
- 待复审
- 复审驳回
- 已通过

#### 8.7.4 验收标准

- 章节分配后对应用户才拥有编辑权
- 提交审批后项目协作页可看到真实审批任务
- 驳回意见可回写到对应章节负责人

---

### 8.8 导出与交付

#### 8.8.1 功能目标

将系统内编制完成的标书按公司规则输出为可交付成品，而非简单文本导出。

#### 8.8.2 功能要求

- 支持导出 Word
- 支持导出 PDF
- 支持模板配置：
  - 封面
  - 标题层级
  - 编号规则
  - 页眉页脚
  - 目录
  - 附件顺序
- 支持导出版本选择：
  - 内部评审版
  - 最终提交版
  - 归档版
- 导出前支持预览目录结构
- 导出记录需保存，便于追溯

#### 8.8.3 验收标准

- 导出结果与系统内章节结构一致
- 目录和标题层级自动生成
- 可按模板导出至少一种正式格式
- 当前实现校准：正式导出文件链路与模板渲染仍需继续收口，现阶段不应将“正式可下载成品文件”视为已完全打通

### 8.9 数据中心

#### 8.9.1 功能目标

沉淀投标数据与复盘结果，反哺未来项目判断、素材推荐和投标策略。

#### 8.9.2 模块要求

1. 看板  
   展示项目数量、中标率趋势、当前项目分布、进度风险

2. 知识库  
   将复盘、案例、优秀表达片段沉淀为可检索知识条目

3. 投标复盘  
   每个项目可填写：
   - 中标 / 落标结果
   - 原因分类
   - 评分拆解
   - 改进项
   - 关键风险
   - 下次复用建议

4. 竞争对手分析  
   建立对手档案，字段包括：
   - 名称
   - 常见报价区间
   - 优势
   - 劣势
   - 常用策略
   - 关联项目

5. 项目归档  
   将最终标书、招标文件、审批记录、复盘、对手分析统一归档

#### 8.9.3 反哺机制

数据中心沉淀的数据应可反哺：

- AI 素材推荐
- 历史标书归类
- 对手策略判断
- 常见失分点提醒

#### 8.9.4 验收标准

- 项目归档后可检索历史项目
- 复盘字段为结构化数据，可用于统计
- 竞争对手信息可关联多个项目

### 8.10 人员配置与系统设置

#### 8.10.1 功能目标

支持管理员维护用户、角色、上下级、模板、模型配置与系统基础参数；当前设置能力拆分为“人员配置”和“模型配置”两个独立入口。

#### 8.10.2 功能要求

- 仅管理员可进入设置相关页面
- 当前导航实现为“人员配置”“模型配置”两个独立入口，不再要求单一综合设置页承载全部内容
- 支持新增、编辑、禁用用户
- 支持角色分配、上级指定、密码重置
- 禁用用户后不可登录、不可参与任务分配
- 至少保留一名管理员
- 支持导出模板维护
- 支持模型 / API 参数配置
- 模型配置页采用单页切换结构，至少包含两个切换视图：
  - API 模型配置
  - Coding Plan 配置
- API 模型配置至少支持以下供应商：
  - OpenAI
  - Claude
  - DeepSeek
  - 豆包
  - 通义千问
  - MiniMax
  - Kimi
  - 智谱
- 支持为单个供应商填写标准 API-Key、保存配置、执行一次真实调用校验并手动生效
- 支持输入 Coding Plan URL + API-Key 并真实解析其兼容地址与支持模型列表
- Coding Plan 仅用于识别兼容地址和支持模型，不直接作为标书解析链路的标准生效密钥
- 支持审计日志查看

#### 8.10.3 验收标准

- 用户禁用后权限立即失效
- 角色和上下级配置可持久化保存
- 管理员可维护至少一套导出模板
- 管理员可完成至少一个标准模型供应商的真实校验与生效
- Coding Plan URL 与 API-Key 可成功解析出支持模型列表

## 九、数据模型

### 9.1 核心实体

| 实体 | 关键字段 |
|------|----------|
| 用户 | 用户名、姓名、角色、状态、上级、联系方式、创建时间 |
| 项目 | 名称、编号、类型、状态、截止时间、负责人、创建时间 |
| 招标文件 | 项目 ID、文件地址、版本、上传人、上传时间 |
| 招标解析结果 | 项目 ID、解析类型、结构化内容、来源定位、确认状态 |
| 应标大纲 | 项目 ID、章节树、版本号、确认状态 |
| 大纲章节 | 章节 ID、父级 ID、标题、层级、排序、章节类型、责任角色 |
| 章节任务 | 项目 ID、章节 ID、负责人、状态、开始时间、截止时间 |
| 标书内容 | 项目 ID、章节 ID、正文、版本号、最后编辑人、更新时间 |
| 企业库资料 | 分类、标题、标签、适用章节、状态、上传人、来源 |
| 片段库条目 | 来源标书、来源章节、摘要、正文、标签、适用场景 |
| 审批记录 | 项目 ID、阶段、审批人、结果、意见、时间 |
| 导出记录 | 项目 ID、模板、导出版本、导出人、导出时间 |
| 复盘记录 | 项目 ID、结果、原因、改进项、得分拆解、建议 |
| 竞争对手 | 名称、特征、价格区间、优势、劣势、关联项目 |
| 归档记录 | 项目 ID、归档内容清单、归档时间、归档人 |

### 9.2 数据关系原则

- 项目是一级业务主键
- 章节任务必须关联具体章节 ID，不可仅关联“技术标 / 商务标”大类
- 历史标书拆解后的片段需保留来源链路
- AI 解析结果与人工修订结果应区分版本

## 十、状态流转与异常场景

### 10.1 项目状态流转

```text
待开始 → 解析中 → 编制中 → 待初审 → 待复审 → 待导出 → 已完成 → 已归档
```

### 10.2 章节状态流转

```text
待开始 → 编写中 → 已完成待提交 → 待初审 → 初审驳回 → 待复审 → 复审驳回 → 已通过
```

### 10.3 异常场景

系统需覆盖以下异常处理：

- 招标文件上传失败或格式不支持
- AI 解析失败或结果不完整
- 用户未选择项目即尝试解析或编制
- 章节已分配但大纲被删除或结构变化
- 禁用用户仍存在未完成任务
- 审批中用户试图再次编辑已锁定章节
- 导出时模板缺失或字段映射错误
- 归档时项目资料不完整

## 十一、非功能需求

### 11.1 性能

- 列表页首屏响应时间目标 2 秒内
- 大文档解析与 AI 生成采用异步任务机制
- 长任务需展示进度、状态与失败重试入口

### 11.2 安全

- 用户登录认证
- 密码加密存储
- 敏感操作需鉴权
- 导出、删除、权限变更等操作需审计留痕

### 11.3 审计

需记录以下关键行为：

- 谁在什么时间修改了哪个章节
- 谁分配了哪个任务
- 谁执行了审批、导出、归档
- AI 结果何时被人工确认或修订

### 11.4 可用性

- 支持现代浏览器：Chrome、Edge、Safari
- 建议分辨率 1280×720 及以上
- 导航、状态、权限反馈需明确可见

### 11.5 可维护性

- AI 输出结构、模板规则、分类标签可配置
- 关键数据模型需支持后续扩展

## 十四、结论

本 PRD v1.1 的核心目标，是把系统从“单点 AI 帮写工具”升级为“以项目为中心、以大纲为纽带、以资料沉淀为资产、以协作和导出为交付保障”的完整标书生产平台。

本版正式将以下四条链路定义为必须打通的产品主链路：

1. 历史标书入库 → 智能归类 → 企业库 / 片段库沉淀  
2. 招标解析 → 应标大纲 → 标书章节导航  
3. 章节编制 → 任务分配 → 审批驳回 → 正式导出  
4. 项目完成 → 归档复盘 → 数据中心反哺  

后续若需继续细化，可在本版基础上拆分以下配套文档：

- 页面原型说明
- 接口清单
- 数据库表设计
- AI Prompt / 输出 Schema 设计
- 测试验收用例

### 16.3 当前前端项目结构

1. 当前前端全局壳层统一收敛在 `src/app/App.tsx`，使用固定左侧导航 + 顶部栏 + `activePage` 状态切换页面，不以路由系统作为主切换方式。

### 16.4 当前后端模块结构

当前与三大真实页面直接相关的后端模块如下：

- `tender`：负责招标文件上传、解析任务创建、解析结果查询、条目来源追踪、原文回看、应标大纲生成。
- `asset`：负责企业库列表、手工上传、下载、删除、历史标书入库任务、条目确认与整单完成。
- `settings`：负责模型配置加载、Coding Plan 解析、供应商连通性校验、配置保存与生效。
- `document`：负责资产 / 招标文件的文档与版本记录，是下载、原文定位和文档追溯的公共底座。
- `storage`：负责 MinIO / 本地存储回退，是文件真实上传下载链路的一部分。
- `rag` / `ai` / `queue`：分别承接结构化抽取、模型调用、异步任务处理，是标书解析和企业库自动入库的支撑模块。

### 16.5 标书解析当前实现逻辑

#### 16.5.1 业务链路

1. 前端在解析前先确保存在远端项目上下文。
2. 招标文件通过 `/tender/upload` 上传，返回远端文件标识。
3. 前端调用 `/tender/parse` 创建解析任务，后端异步处理。
4. 前端轮询 `/tender/parse/status/:taskId`，持续获取：
   - 整体进度
   - 当前阶段
   - 7 个主类的分类进度
   - 已完成主类的部分解析结果
5. 单条解析结果支持通过 `/tender/parse/items/:itemId/trace` 查看来源块、页码、章节路径和原文引用。
6. 需要回看原始文件时，通过 `/tender/documents/:documentVersionId/source-file` 获取原文文件进行预览。

#### 16.5.2 当前页面能力

- 当前页面主展示 7 个主类：
  - 基础信息
  - 资格要求
  - 评审要求
  - 投标文件要求
  - 无效标与废标项
  - 应标需提交文件
  - 招标文件审查
- 每个主类具有独立状态：`pending / running / completed / failed`
- 已完成主类可先展示、先查看，不阻塞其他分类继续解析
- 条目支持查看来源证据，满足“AI 结果可追溯”的产品要求

#### 16.5.3 当前范围说明

1. 标书解析当前是“真实任务式解析页”，不是静态示意页。
2. 当前前端重点闭环在“上传 → 解析 → 分类结果 → 来源追踪”。
3. 后端虽然已提供 `/tender/outline/generate`，但“解析结果自动驱动标书编制页章节导航”的完整闭环仍不按当前真实落地页面计入。

### 16.6 企业库当前实现逻辑

#### 16.6.1 页面结构

企业库当前由两类页面状态组成：

1. `标书入库`
   - 上传历史 `DOCX`
   - 创建入库任务
   - 查看任务列表
   - 查看任务结果
   - 条目级“正式入库”
   - 整单“确认本标书已正式入库”
2. 分类资料页
   - 企业资质
   - 项目业绩
   - 技术方案
   - 标书归档
   - 中标案例
   - 人员简历

#### 16.6.2 自动入库链路

1. 历史标书入库当前仅接受 `DOCX`，目的是尽量保持拆分后文档的格式、表格与字体一致。
2. 前端通过 `/assets/ingest/jobs` 创建入库任务，并轮询任务列表与任务详情。
3. 后端自动拆分出候选条目，状态分为：
   - `completed`
   - `pending_review`
4. 对于待确认条目，用户可手工指定分类、子类和标题，再执行“正式入库”。
5. 当整份标书的可入库内容确认完成后，用户可点击“确认本标书已正式入库”，该任务会从当前活跃任务列表中清空。
6. 删除入库任务时，需要同步删除：
   - 对应任务结果
   - 对应下载文件
   - 已由该任务生成的企业库资产

#### 16.6.3 手工上传链路

1. 当前企业库六大分类均支持手工上传。
2. 企业资质页按六个子类卡片展示，并支持逐卡上传：
   - 公司基本情况表
   - 公司简介
   - 公司资质清单
   - 客户占有率
   - 与监管合作案例
   - 公司组织架构
3. 项目业绩、技术方案、标书归档、中标案例、人员简历当前以表格列表展示，支持上传、下载、删除。
4. 手工上传不再是纯前端假提示，而是通过 `/assets` 进入真实资产存储链路，并通过文档版本能力提供下载。

#### 16.6.4 当前范围说明

1. 企业库当前已不应再写成“历史标书智能归类仍以占位为主”。
2. 当前应描述为：“企业库已具备真实入库、真实手工上传、真实下载删除；但语义提取准确率与复杂分类精度仍需继续增强。”
3. 企业资质页面当前按两列卡片排版，不应继续按旧设计文档中的三列常驻结构描述。

### 16.7 模型配置当前实现逻辑

#### 16.7.2 API 模型配置能力
