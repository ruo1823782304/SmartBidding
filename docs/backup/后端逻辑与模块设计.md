# 智能标书库 · 后端逻辑与模块设计

**版本**：v1.0  
**依据**：PRD-智能标书库.md、接口表与字段说明.md、技术选型.md  
**目的**：按模块划分后端逻辑，便于定位问题与迭代。

---

## 一、整体结构

### 1.1 技术栈

- **框架**：NestJS (Node.js + TypeScript)
- **数据库**：PostgreSQL
- **ORM**：Prisma
- **缓存/队列**：Redis（可选）+ BullMQ（解析/导出等异步）
- **存储**：MinIO 或 OSS/S3（招标文件、企业库文件、导出文件）
- **文档**：Swagger (OpenAPI)

## 二、公共层（common）

### 2.1 职责

- 全局前缀 `/api`
- JWT 校验：`JwtAuthGuard`，从 token 解析出 `username`，注入 `@CurrentUser()`
- 角色校验：`@Roles('管理员')` + `RolesGuard`，仅管理员可访问 admin、settings
- 统一异常：`HttpExceptionFilter`，返回 `{ code, message }`
- 统一响应（可选）：`{ data, message }` 包装
- 分页：`PageDto`（page, pageSize）、`PaginatedResult<T>`

### 2.2 关键文件

| 文件 | 说明 |
|------|------|
| `common/guards/jwt-auth.guard.ts` | 使用 JwtStrategy 校验 Bearer token |
| `common/guards/roles.guard.ts` | 校验当前用户 role 是否在 @Roles() 中 |
| `common/decorators/current-user.decorator.ts` | 取 req.user（由 JwtStrategy 注入） |
| `common/decorators/roles.decorator.ts` | 元数据 roles[] |
| `common/filters/http-exception.filter.ts` | 格式化 4xx/5xx |

### 2.3 问题定位

- **401 未登录**：查 JwtModule 配置、JwtStrategy、前端是否带 Authorization
- **403 无权限**：查 RolesGuard、当前用户 role、接口是否误加 @Roles

## 三、模块 1：认证（auth）

### 3.1 职责

- 登录：校验用户名密码，签发 JWT（及可选 RefreshToken）
- 登出：可选使 refreshToken 失效或仅前端删 token

### 3.2 接口与路径

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/auth/login | 登录，返回 token + user |
| POST | /api/auth/logout | 登出（可选） |

### 3.3 目录与文件

```
src/auth/
├── auth.module.ts
├── auth.controller.ts    # login, logout
├── auth.service.ts       # validateUser, login → token
├── dto/
│   ├── login.dto.ts      # username, password, remember?
│   └── login-response.dto.ts  # token, user
├── strategies/
│   └── jwt.strategy.ts   # validate(payload) → user，供 Guard 使用
└── auth.module 需导入 User 模块（查用户）、JwtModule
```

### 3.4 依赖数据

- **User 表**：通过 Prisma 查用户（username、password 需 bcrypt 比对）、状态为「启用」才允许登录。

### 3.5 核心逻辑

- `AuthService.login(dto)`：按 username 查用户 → 校验 status=启用 → bcrypt.compare(password) → 生成 JWT（payload 含 username、role）→ 返回 token + UserInfo（不含密码）。
- `JwtStrategy.validate(payload)`：用 payload.username 再查一次用户，返回 { username, role } 等，供 @CurrentUser() 使用。

### 3.6 问题定位

- **登录 401**：AuthService 校验逻辑、用户是否存在、status、密码是否加密一致。
- **其他接口 401**：JwtStrategy、token 过期时间、前端传参。

### 4.1 职责

- 获取当前用户信息（/api/user/me）
- 更新个人资料与密码（/api/user/profile）

### 4.2 接口与路径

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/user/me | 当前用户信息 |
| PUT | /api/user/profile | 更新 name、contact、newPassword |

### 4.3 目录与文件

```
src/user/
├── user.module.ts
├── user.controller.ts   # me, updateProfile
├── user.service.ts      # findByUsername, updateProfile
└── dto/
    └── update-profile.dto.ts  # name?, contact?, newPassword?
```

### 4.4 依赖数据

- **User 表**：name, contact, password（更新时 bcrypt  hash 新密码）。

### 4.5 核心逻辑

- `me()`：从 @CurrentUser() 取 username，查库返回 UserInfo（不含 password）。
- `updateProfile(username, dto)`：只更新当前用户；若有 newPassword 则校验后 hash 写入。

### 4.6 问题定位

- **资料不更新**：UserService.updateProfile、DTO 字段、Prisma update 条件（where username）。

### 5.1 职责

- 用户列表、新增、编辑、重置密码、禁用/启用
- 仅管理员可访问（@Roles('管理员')）

### 5.2 接口与路径

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/admin/users | 用户列表 |
| POST | /api/admin/users | 新增用户 |
| PUT | /api/admin/users/:username | 编辑用户 |
| POST | /api/admin/users/:username/reset-password | 重置密码 |
| PATCH | /api/admin/users/:username/status | 禁用/启用 |

### 5.3 目录与文件

```
src/admin/
├── admin.module.ts
├── admin-users.controller.ts
├── admin-users.service.ts
└── dto/
    ├── create-user.dto.ts
    ├── update-user.dto.ts
    ├── reset-password.dto.ts
    └── user-status.dto.ts
```

### 5.4 依赖数据

- **User 表**：username, name, password, role, superior, status, createdAt 等。
- **约束**：用户总数 ≤20；至少 1 名管理员；用户名唯一；禁用后不可登录（在 Auth 里校验）。

### 5.5 核心逻辑

- 新增：校验总数、用户名唯一、密码 hash、默认 status=启用。

- 重置密码：仅管理员可操作，新密码 hash 后写入。

### 5.6 问题定位

- **新增失败**：总数、唯一约束、DTO 校验。
- **禁用/编辑失败**：至少一名管理员约束、RolesGuard。

## 七、模块 5：标书解析（tender）

### 7.1 职责

- 上传招标文件（存对象存储，落库记录）
- 发起解析（异步任务，入队）
- 查询解析进度/结果（轮询或 SSE）
- 生成应标大纲（可调 AI，结果写入 project outline）

### 7.2 接口与路径

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/tender/upload | 上传文件，绑定 projectId |
| POST | /api/tender/parse | 发起解析，返回 taskId |
| GET | /api/tender/parse/result 或 /api/tender/parse/status/:taskId | 进度与结果 |
| POST | /api/tender/outline/generate | 生成应标大纲 |

### 7.3 目录与文件

```
src/tender/
├── tender.module.ts
├── tender.controller.ts
├── tender.service.ts         # upload, createParseJob, getParseResult, generateOutline
├── tender-parse.processor.ts # Bull 消费者：解析 PDF/Word，写 TenderParseResult
├── dto/
│   ├── upload-tender.dto.ts  # projectId (form)
│   ├── parse-tender.dto.ts
│   └── parse-result.dto.ts   # 与前端读标结果结构一致
└── entities/ 或 prisma 模型
    # TenderFile: id, projectId, fileId, fileName, size, createdAt
    # TenderParseTask: id, projectId, fileId, status, progress, result(json), createdAt
```

### 7.4 依赖数据

- **TenderFile**：项目绑定、文件存储 key、文件名、大小。
- **TenderParseTask**：status(pending/running/done)、progress、result(JSON，对应前端标签页结构)。
- **Project**：写 outline 时更新 project.outline。

### 7.5 核心逻辑

- **upload**：multipart 存 MinIO/OSS，写 TenderFile 表，返回 fileId。
- **parse**：创建 TenderParseTask，入队 Bull；Processor 内：读文件 → 解析（pdf-parse/mammoth）→ 可选调 AI 抽取结构化结果 → 写 task.result、status=done。
- **getParseResult**：按 projectId 或 taskId 查 TenderParseTask，返回 status、progress、result。
- **generateOutline**：可基于 parse result 或默认模板，可调 LLM；结果写入 Project outline（techOutlineSections、bizOutlineSections）。

### 7.6 问题定位

- **上传失败**：存储配置、multipart 解析、projectId 校验。
- **解析一直 pending**：队列是否消费、Processor 是否报错、文件是否可读。
- **大纲不生成**：ProjectService 更新 outline、AI 配置与超时。

## 八、模块 6：标书编制（proposal）

### 8.1 职责

- 章节内容的增删改查、标记完成
- 按章节推荐素材（查企业库）
- 提交审批（改项目/标书状态，可触发任务状态）
- 导出标书（Word/PDF，异步可入队）

### 8.2 接口与路径

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/projects/:projectId/sections/:sectionKey/content | 章节内容 |
| PUT | /api/projects/:projectId/sections/:sectionKey/content | 保存内容 |
| POST | /api/projects/:projectId/sections/:sectionKey/complete | 标记完成 |
| GET | /api/projects/:projectId/sections/:sectionKey/recommendations | 推荐素材 |
| POST | /api/projects/:projectId/proposal/submit | 提交审批 |
| GET 或 POST | /api/projects/:projectId/proposal/export | 导出 |

### 8.3 目录与文件

```
src/proposal/
├── proposal.module.ts
├── section.controller.ts    # sections/:sectionKey/content, complete, recommendations
├── proposal.controller.ts  # proposal/submit, proposal/export
├── section.service.ts       # getContent, saveContent, setComplete, getRecommendations
├── proposal.service.ts      # submit, export（组章节内容、调 docx 生成）
└── dto/
    ├── section-content.dto.ts
    ├── section-complete.dto.ts
    └── export-format.dto.ts
```

### 8.4 依赖数据

- **SectionContent**：projectId, sectionKey, content(text), version, completed(boolean), lastEditedAt, lastEditedBy。
- **Asset**：推荐时按 category 等查询（见 asset 模块）。
- **Project**：提交审批时更新 proposalStatus 或 workflow 状态；导出时读 outline + 所有 section 内容。

### 8.5 核心逻辑

- **content**：sectionKey 即「group::sectionName」；保存时校验当前用户是否有编辑权（查 section-assignment 或角色）。
- **complete**：更新 SectionContent.completed。
- **recommendations**：按 sectionKey 推断类型（资信/商务/技术等），查 Asset 表推荐列表。
- **submit**：更新项目状态为「待初审」等；可同时创建/更新协作任务「标书审核任务」为进行中（见 task 模块）。
- **export**：按 outline 顺序拉取所有 section 内容，组装 Word（docx）或生成 PDF；大文件可异步队列 + 返回 taskId，再提供「下载结果」接口。

### 8.6 问题定位

- **无权限保存**：SectionService 权限校验、SectionAssignment 模块数据。
- **提交/导出失败**：ProposalService、项目状态、任务联动、文件生成与存储。

## 十、模块 8：协作任务与审批（task）

### 10.1 职责

- 任务列表（按项目、状态、myOnly 筛选）
- 分配任务（含标书制作人员分配时写 sectionKeys → section-assignment）
- 审批通过/驳回
- 审批记录列表

### 10.2 接口与路径

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/projects/:projectId/tasks | 任务列表，支持 status、myOnly |
| POST | /api/tasks/:taskId/assign | 分配 |
| POST | /api/tasks/:taskId/approve | 通过 |
| POST | /api/tasks/:taskId/reject | 驳回 |
| GET | /api/projects/:projectId/approvals | 审批记录 |

### 10.3 目录与文件

```
src/task/
├── task.module.ts
├── task.controller.ts
├── task.service.ts          # list, assign, approve, reject, listApprovals
└── dto/
    ├── assign-task.dto.ts   # username, roleName, sectionKeys?
    ├── approve.dto.ts
    └── reject.dto.ts        # reason
```

### 10.4 依赖数据

- **Task**：id, projectId, name, taskType, requirement, urgency, sectionKey, status, createdAt 等。
- **TaskAssignment**：taskId, username, roleName, assignedAt, progress。
- **TaskHistory**：taskId, at, status, note, operator。
- **ApprovalRecord**（可单独表或从 TaskHistory 筛）：projectId, taskId, operator, action(通过/驳回), comment, at。

### 10.5 核心逻辑

- **list**：按 projectId 查 Task，可选 status；myOnly 时筛 assignments 包含当前 username 的任务。
- **assign**：校验角色有分配权限；写 TaskAssignment、TaskHistory；若为「标书制作人员分配」且传了 sectionKeys，调用 SectionAssignmentService 写入章节分配。
- **approve/reject**：更新 task.status、写 TaskHistory/ApprovalRecord；驳回时 reason 写入 note。
- **listApprovals**：按 projectId 查审批记录表或 TaskHistory 中 action 为通过/驳回的条目。

### 10.6 问题定位

- **任务列表为空**：projectId、status 筛选、myOnly 与当前用户。
- **分配后章节无权限**：是否调了 SectionAssignmentService、sectionKeys 与前端一致。
- **审批不生效**：Task 状态枚举、History 写入、前端调用的 taskId。

- **batch-classify**：接收多文件，入队；Processor 内可调 AI 归类，结果写回 Asset 或单独「归类结果」表（当前可仅返回 success + taskId，前端示意）。

### 11.6 问题定位

- **列表/上传问题**：category 枚举、存储路径、分页参数。
- **推荐素材来自本模块**：Proposal 模块调 AssetService 或直接查 Asset 表。

### 16.4 auth.service.ts（核心逻辑）

```ts
import { Injectable, UnauthorizedException } from '@nestjs/common';
import { JwtService } from '@nestjs/jwt';
import * as bcrypt from 'bcrypt';
import { PrismaService } from '../prisma.service'; // 或 UserService
import { LoginDto } from './dto/login.dto';

@Injectable()
export class AuthService {
  constructor(
    private readonly prisma: PrismaService,
    private readonly jwt: JwtService,
  ) {}

  async login(dto: LoginDto) {
    const user = await this.prisma.user.findUnique({ where: { username: dto.username } });
    if (!user || user.status !== '启用') throw new UnauthorizedException('用户名或密码错误');
    const ok = await bcrypt.compare(dto.password, user.password);
    if (!ok) throw new UnauthorizedException('用户名或密码错误');
    const token = this.jwt.sign({ sub: user.id, username: user.username, role: user.role });
    const userInfo = { username: user.username, name: user.name, role: user.role, contact: user.contact ?? undefined };
    return { token, user: userInfo };
  }
}
```

### 16.5 strategies/jwt.strategy.ts（供 Guard 使用）

```ts
import { Injectable } from '@nestjs/common';
import { PassportStrategy } from '@nestjs/passport';
import { ExtractJwt, Strategy } from 'passport-jwt';
import { PrismaService } from '../../prisma.service';

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy, 'jwt') {
  constructor(private readonly prisma: PrismaService) {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: process.env.JWT_SECRET || 'your-secret',
    });
  }

  async validate(payload: { sub: string; username: string; role: string }) {
    const user = await this.prisma.user.findUnique({ where: { id: payload.sub } });
    if (!user || user.status !== '启用') return null;
    return { userId: user.id, username: user.username, role: user.role };
  }
}
```

### 16.6 DTO 示例

- **login.dto.ts**：`username: string`，`password: string`，`remember?: boolean`（可选延长过期时间）。
- **login-response.dto.ts**：`token: string`，`user: { username, name, role, contact? }`。

其他模块（user、admin、project、tender、proposal、section-assignment、task、asset、dashboard、settings）按同一模式：**Controller 只做路由与 DTO 校验 → Service 做业务与 Prisma 调用 → 需要鉴权则加 `@UseGuards(JwtAuthGuard)` 与 `@Roles()`**。
