## 二、产品定位与目标

### 2.1 产品定位

智能标书库是一套面向中小型投标团队的标书全生命周期管理系统，覆盖以下完整链路：

**资料沉淀 → 招标解析 → 应标大纲生成 → 标书编制 → 协作审批 → 导出交付 → 归档复盘**

系统的核心不是单点“AI 写标书”，而是围绕“项目”和“应标大纲”建立一套可持续复用、多人协作、可导出交付、可复盘沉淀的标书生产平台。

### 2.2 核心价值

| 价值点 | 说明 |
|------|------|
| 招标即大纲 | 招标文件解析后直接生成应标大纲，减少人工拆目录 |
| 资料可复用 | 历史标书、资质、简历、业绩、技术方案可持续沉淀 |
| AI 辅助写作 | AI 负责提取、归类、推荐、生成草稿，人工做确认与修订 |
| 多人可协作 | 章节级任务分配、权限控制、初审复审和驳回闭环 |
| 输出可交付 | 按公司模板、标题规则、目录结构导出 Word / PDF |
| 数据可复盘 | 项目归档、投标复盘、竞争对手档案反哺后续投标 |

### 2.3 业务目标

1. 缩短从收到招标到形成标书初稿的时间  
2. 提高历史资料和表达片段的复用率  
3. 让多人协作过程中职责更清晰、进度更透明  
4. 降低最终导出成品的人工排版和返工成本  
5. 建立投标知识资产，为后续项目复用和策略判断服务  

### 2.4 成功指标

| 指标 | 目标建议 |
|------|----------|
| 招标解析后生成首版应标大纲时长 | 5 分钟内 |
| 历史资料 / 片段复用率 | ≥ 40% |
| AI 生成大纲一次采纳率 | ≥ 70% |
| 导出后人工排版调整时长 | 降低 50% |
| 章节任务按时完成率 | ≥ 85% |
| 项目归档完成率 | ≥ 95% |
| 复盘完成率 | ≥ 80% |

## 四、目标用户与角色

### 4.1 目标用户

- 投标负责人 / 项目经理
- 技术标制作人
- 商务标制作人
- 机动人员
- 初审人
- 复审人
- 管理员

### 4.2 角色定义

| 角色 | 核心职责 |
|------|----------|
| 管理员 | 用户、角色、上下级、模板、模型配置、系统设置、全局查看 |
| 项目经理 | 创建项目、分配章节任务、追踪进度、推动审批 |
| 技术标制作人 | 编写技术标相关章节 |
| 商务标制作人 | 编写商务标、资信标相关章节 |
| 机动人员 | 协助资料上传、辅助章节编制 |
| 初审人 | 对提交版本执行初审，可通过或驳回 |
| 复审人 | 对初审通过版本执行终审，可通过或驳回 |

### 4.3 权限原则

- 权限分两层：
  - 页面级权限：由角色决定
  - 章节级权限：由角色 + 是否被分配章节共同决定
- 初审人 / 复审人默认只读查看标书内容，不直接改正文
- 管理员 / 项目经理有全局查看和协调能力
- 禁用用户后不可登录、不可参与任务分配、不可继续编辑

## 六、核心业务流程

### 6.1 总流程

```text
资料沉淀
  ├─ 上传资质 / 业绩 / 技术方案 / 简历
  └─ 历史标书批量入库 → AI 拆解归类 → 写入企业库 / 片段库

收到新招标
  ├─ 创建项目
  ├─ 上传招标文件
  ├─ AI 解析招标要求
  └─ 生成应标大纲 → 人工微调确认

标书编制
  ├─ 章节导航 = 项目应标大纲
  ├─ 按章节分配负责人
  ├─ 按章节复用历史资料 / AI 生成草稿
  ├─ 编辑、保存、版本记录
  └─ 全部完成后提交审批

协作审批
  ├─ 初审
  ├─ 驳回后回到对应章节负责人
  ├─ 复审
  └─ 审批通过后进入导出

导出与归档
  ├─ 选择模板、编号规则、导出版本
  ├─ 导出 Word / PDF
  ├─ 填写复盘
  └─ 归档项目资料并沉淀为知识资产
```

### 6.2 关键设计原则

- 以“项目”为主线，不做跨项目混乱编辑
- 以“应标大纲”为编制导航和任务分配依据
- AI 结果必须可确认、可修订、可追溯
- 历史沉淀必须能在后续项目中被检索和推荐

## 七、AI 设计

### 7.1 AI 总原则

- AI 只做辅助，不直接替代最终判断
- 结构化输出优先于长文本输出
- 关键 AI 结果需要人工确认
- 有结论的地方尽量保留来源定位
- 人工修订结果可回写系统，作为后续推荐依据

### 7.2 AI 场景清单

| 场景 | 输入 | 输出 | 是否需人工确认 |
|------|------|------|----------------|
| 招标文件解析 | 招标文件全文 | 资格要求、评分点、废标条款、提交材料、风险点 | 是 |
| 应标大纲生成 | 解析结果 + 公司模板偏好 | 一/二/三级章节树、章节类型、建议负责人 | 是 |
| 历史标书智能归类 | 多份历史标书 | 共性维度、分类结果、片段拆解、标签 | 是 |
| 资质 / 简历抽取 | 资质文件、简历文件 | 字段信息、标签、建议适用章节 | 是 |
| 章节素材推荐 | 当前章节 + 项目要求 | 推荐资质、业绩、简历、方案、历史片段 | 插入前确认 |
| 章节草稿生成 | 当前章节要求 + 素材 | 章节草稿正文 | 是 |
| 复盘摘要生成 | 项目结果 + 审批记录 + 人工输入 | 复盘摘要、改进建议 | 是 |

### 7.3 AI 输出要求

- 招标解析需输出结构化字段，不只是普通摘要
- 应标大纲必须是结构化章节树，不只是标题列表
- 历史标书归类结果需能落到企业库分类或片段库
- 推荐素材需要带来源标识
- AI 草稿生成应支持选择语气、长度、引用素材范围

## 八、模块设计

### 8.1 项目管理

#### 目标

管理每一次投标任务，并作为所有业务数据的统一容器。

#### 核心功能

- 创建项目
- 编辑项目信息
- 切换当前项目
- 查看项目看板
- 更新项目状态
- 归档项目

#### 核心字段

- 项目名称
- 项目编号
- 招标单位
- 项目类型
- 截止时间
- 项目负责人
- 当前状态

#### 状态建议

- 待开始
- 解析中
- 编制中
- 待初审
- 待复审
- 待导出
- 已完成
- 已归档

---

### 8.2 标书解析

#### 目标

把原始招标文件转为结构化、可追溯、可用于后续写作的大纲基础数据。

#### 核心功能

- 上传招标文件
- 触发 AI 解析
- 查看解析进度
- 查看结构化解析结果
- 编辑解析结果
- 基于解析结果生成应标大纲

#### 解析结果建议结构

- 资格要求
- 评分标准
- 投标文件组成
- 废标 / 无效条款
- 关键时间节点
- 特殊格式要求
- 风险提示
- 来源定位

#### 设计重点

- 解析结果要清晰分组
- 每一条结论要支持看来源
- 页面要明确区分“原始招标要求”与“系统建议”

---

### 8.3 应标大纲管理

#### 目标

形成贯穿解析、编制、任务分配、导出的统一章节结构。

#### 核心功能

- 生成应标大纲
- 支持三级章节
- 编辑章节标题
- 增删章节
- 拖拽排序
- 调整层级
- 同步到标书编制导航和导出目录

#### 章节节点建议字段

- 章节 ID
- 父级 ID
- 标题
- 层级
- 排序
- 章节类型
- 是否必填
- 建议负责人角色

#### 设计重点

- 大纲编辑后要提示影响范围
- 已被分配的章节被删除或重命名时要有确认机制
- 要区分“默认模板”和“AI 生成大纲”

---

### 8.4 标书编制

#### 目标

基于项目应标大纲逐章编写标书，并尽量复用历史资料和 AI 草稿。

#### 核心功能

- 左侧章节导航
- 章节正文编辑
- 从企业库 / 历史片段复用
- AI 生成草稿
- 自动推荐素材
- 保存与自动保存
- 历史版本查看与恢复
- 标记章节完成
- 提交审批
- 全书预览

#### 设计重点

- 左侧导航必须与项目大纲一致
- 每章节必须有“历史复用”和“AI 生成”两个清晰入口
- 要区分“可编辑章节”和“只读章节”
- 要能看到章节当前状态、负责人、最后编辑时间

#### 协作规则

- 用户只可编辑自己被分配的章节
- 同一章节默认单人编辑
- 审批中章节默认锁定
- 驳回可指定退回整书或指定章节

---

### 8.5 企业库

#### 目标

统一沉淀投标资料，作为推荐和复用的核心数据源。

#### 分类

- 企业资质
- 项目业绩
- 技术方案
- 中标案例
- 人员简历
- 标书归档
- 历史片段库

#### 核心功能

- 上传文件 / 录入文本
- 分类管理
- 标签管理
- 字段化抽取
- 资料状态管理
- 适用章节关联
- 搜索与筛选

#### 设计重点

- 企业库不只是文件堆放，要有可推荐、可筛选、可引用的结构
- 要区分“原始资料”“抽取字段”“推荐片段”

---

### 8.6 历史标书入库与智能归类

#### 目标

把历史标书从“整份文件”变成“可拆解、可归类、可检索、可复用”的资产。

#### 核心功能

- 批量上传历史标书
- AI 识别目录结构
- 拆解章节与片段
- 提取共性维度
- 给出归类建议
- 写入企业库对应分类
- 形成历史片段库
- 人工确认和修正归类结果

#### 共性维度建议

- 资质类
- 业绩类
- 人员类
- 技术方案类
- 商务承诺类
- 格式模板类

#### 设计重点

- 必须显示“来源于哪份标书、哪一章节”
- 必须支持人工纠正归类
- 归类结果要能直接用于后续章节推荐

---

### 8.7 项目协作与审批

#### 目标

将标书编制从个人写作变成多人协作、责任清晰、流程可控的交付过程。

#### 核心功能

- 项目看板
- 章节任务分配
- 我的任务
- 我负责的章节
- 待办审批
- 初审 / 复审
- 审批记录

#### 章节状态建议

- 待开始
- 编写中
- 已完成待提交
- 待初审
- 初审驳回
- 待复审
- 复审驳回
- 已通过

#### 设计重点

- 章节分配必须落库，不能只保留前端状态
- 驳回意见要回写到对应章节或整书
- 个人中心要能一键进入“我负责的章节”

---

### 8.8 导出与交付

#### 目标

把系统内的章节内容输出为正式可交付的标书成品，而不是普通文本。

#### 核心功能

- 导出 Word
- 导出 PDF
- 选择公司模板
- 设置标题层级与编号规则
- 生成目录
- 选择封面、页眉页脚、附件顺序
- 导出前预览
- 导出记录留痕

#### 导出版本建议

- 内部评审版
- 最终提交版
- 归档版

#### 设计重点

- 目录结构要与应标大纲一致
- 导出结果要有“模板”“版本”“时间”“导出人”信息

---

### 8.9 数据中心

#### 目标

沉淀项目结果、复盘经验、竞争对手信息，并反哺后续投标。

#### 子模块

1. 看板  
   项目数量、中标率趋势、当前状态分布、进度风险

2. 知识库  
   复盘、案例、优秀片段沉淀为知识条目

3. 投标复盘  
   中标 / 落标、原因、评分拆解、改进项、风险点

4. 竞争对手分析  
   对手名称、价格区间、优势、劣势、常用策略、关联项目

5. 项目归档  
   最终标书、招标文件、审批记录、复盘、对手分析统一归档

#### 设计重点

- 复盘必须结构化，不要只留自由文本
- 归档后的项目要能参与后续历史标书检索和推荐

---

### 8.10 人员配置与系统设置

#### 目标

让管理员完成人员、角色、上下级、模板和模型配置。

#### 核心功能

- 用户列表
- 新增用户
- 编辑用户
- 重置密码
- 禁用 / 启用用户
- 设置上下级关系
- 维护模型 / API 配置
- 维护导出模板
- 查看审计日志

#### 设计重点

- 设置入口建议仅管理员可见
- 禁用用户后应立即失效

- 右侧：素材推荐 / AI 生成 / 审批意见

## 十、关键交互与状态

### 10.1 项目状态

```text
待开始 → 解析中 → 编制中 → 待初审 → 待复审 → 待导出 → 已完成 → 已归档
```

### 10.2 章节状态

```text
待开始 → 编写中 → 已完成待提交 → 待初审 → 初审驳回 → 待复审 → 复审驳回 → 已通过
```

### 10.3 关键权限判断

某章节是否可编辑 = 是否有编制权限 + 是否被分配该章节 + 当前章节未锁定 + 当前项目处于可编辑状态

### 10.4 异常场景

设计时必须考虑以下异常反馈：

- 未选项目就进入解析或编制
- AI 解析失败
- 大纲编辑影响已分配任务
- 已禁用用户仍有未完成任务
- 审批中重复编辑同一章节
- 导出时模板缺失
- 归档时资料不完整

## 十五、当前未打通与设计注意事项

以下问题已经在原始文档中暴露，本次统一纳入设计注意项。

### 15.1 数据流未完全打通

- 章节任务分配后，权限未必真正写入持久化数据
- 招标解析结果到结构化应标大纲的自动转换仍需补强
- 首页进入项目后，当前项目上下文需要显式同步
- 提交审批后需真正生成审批任务，不只是提示

### 15.2 设置与用户管理未闭环

- 编辑用户、重置密码需要真实流程，不是占位提示
- 禁用用户后必须在登录和权限层都生效

### 15.3 个人中心跳转体验未闭环

- “我负责的章节”应直接定位到对应章节
- “待办审批”应自动筛选当前用户待办

### 15.4 企业库与归类结果未打通

- 历史标书归类结果必须写入企业库或片段库
- 项目归档状态与看板状态、真实项目列表需要联动

### 15.5 审批与只读预览需明确

- 初审 / 复审是否进入标书编制页，建议提供只读预览能力
- 驳回是退回整书还是指定章节，必须在产品上定义清楚

### 15.6 大纲与权限绑定需保持一致

- 大纲变化会影响章节导航、任务分配、权限、导出目录
- 后续设计必须把这几者看作同一结构的不同视图
